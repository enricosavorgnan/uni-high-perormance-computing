# Inspect Manual vs. Compiler Optimization

**Context:** Incremental optimization of a nested 3D grid distance-checking loop.
**Compiler:** GCC 14.2.0 (`-O3 -march=alderlake -mtune=alderlake`)

## Overview

Following-up the lecture on trivial inefficiencies, the goal of this analysis is to determine how well, **in a very simple case**, the compiler optimizes a naive implementation (`loop.1.c`) compared to a manually optimized version (`loop.2.c`) when aggressive `-O3` optimizations are enabled.

In `loop.2.c`, three major manual optimizations were introduced:

1. **Replacing** `pow(..., 2)**` with explicit multiplication `(val * val)`.
2. **Loop Hoisting:** Extracting the invariant computations for the `i` and `j` coordinates out of the innermost `k` loop.
3. **Reciprocal Multiplication:** Replacing the costly floating-point division `(double)k / Ng` with multiplication by a precomputed inverse `k * Ng_inv`.

I provide the assembly generated by GCC 14.2.0 on an Intel’s `alderlake` architecture with options `-O3 -march=native -mtune=native`  
By examining the assembly (`loop.1.s` and `loop.2.s`), we can check which of the three optimizations the compiler was able to figure out on its own, and which required human intervention.

---

## 1. The `pow()` Function Call

**Did the compiler need our intervention? No.**

In `loop.1.c`, the code calls `pow(..., 2.0)` and `pow(..., 2)`, while `loop.2.c` manually expands this to `x * x`.

If we look at `loop.1.s`, there is **no `call pow` instruction anywhere**.
GCC is smart enough to recognize `pow(val, 2)` and internally translates it into a simple multiplication.

* In `loop.1.s` (line ~164), we see `vmulsd xmm7, xmm7, xmm7` and subsequent `vfmadd132sd` (Fused Multiply-Add) instructions. The compiler successfully squared the values natively.

*Takeaway:* At `-O3`, modern compilers reliably optimize `pow(x, 2)` to `x * x` without manual expansion.

## 2. Loop Hoisting (Loop Invariant Code Motion)

**Did the compiler need our restructuring? No.**

In `loop.1.c`, the entire `dist` calculation is written inside the innermost `k` loop, even though the `i` and `j` terms only change in the outer loops. In `loop.2.c`, we explicitly pre-calculated `dx2` and `dy2` in their respective loops.

GCC's **Loop Invariant Code Motion (LICM)** optimization has spotted this.
If we look into the assembly in `loop.1.s`:

* **The `i` loop (`.L17`):** Computes `powmult_139` (which is `dx2`).
* **The `j` loop (`.L16`):** Computes `_36` (which is `dx2 + dy2`).
* **The `k` loop (`.L13`):** Only computes the `z` distance and adds it to the precomputed `_36`.

*Takeaway:* The compiler easily restructured the math to match the hoisted layout of `loop.2.c`. Writing the hoisted code manually is excellent practice for readability and complexity awareness, but at `-O3`, the compiler can managed that.

## 3. Division vs. Reciprocal Multiplication

**Did the compiler need our gotcha? YES. This is the critical difference.**

In `loop.1.c`, the grid coordinates are calculated using division: `(double)k / Ng`.
In `loop.2.c`, this is replaced by multiplication: `k * Ng_inv`.

Because floating-point math is not associative and strict IEEE 754 rules apply, **the compiler is strictly forbidden from converting `a / b` to `a * (1.0 / b)`**, even at `-O3`. The rounding errors between the two operations differ slightly, so the compiler must do exactly what the C code asks.

This results in a performance difference visible in the assembly:

**In `loop.1.s` (The `k` Loop at `.L13`):**

```assembly
.L13:
    vcvtsi2sd xmm0, xmm3, edx     # Convert integer 'k' to double
    vdivsd    xmm0, xmm0, xmm4    # <--- EXPENSIVE DIVISION (k / Ng)
    vsubsd    xmm0, xmm8, xmm0    
    vaddsd    xmm0, xmm0, xmm5    
    vfmadd132sd xmm0, xmm2, xmm0  # Square and add

```

Every single iteration of the inner loop executes a `vdivsd` instruction. Floating-point division is notoriously slow, taking roughly **10 to 15 clock cycles** of latency.

**In `loop.2.s` (The `k` Loop at `.L12`):**

```assembly
.L12:
    vcvtsi2sd xmm0, xmm3, eax     # Convert integer 'k' to double
    vfnmadd132sd xmm0, xmm7, xmm5 # <--- FAST FUSED MULTIPLY-ADD
    vaddsd    xmm0, xmm0, xmm4    
    vfmadd132sd xmm0, xmm2, xmm0  # Square and add

```

Of course, here the `vdivsd` is absent. Instead, the compiler uses `vfnmadd132sd` (Fused Negative Multiply-Add). It multiplies `k` by the precomputed `Ng_inv` (stored in register `xmm5`) and performs a subtraction in a single, hardware-optimized step. An FMA instruction typically has a latency of only **4 clock cycles** and is fully pipelined.

*Takeaway:* Because of IEEE 754 compliance, the compiler *cannot* optimize division into reciprocal multiplication without the `-freciprocal-math` flag. **This manual optimization in `loop.2.c` is responsible for a massive reduction in CPU cycles.**

## Conclusion

While aggressive compiler optimizations (`-O3`) are incredibly capable of unrolling, hoisting, and replacing naive library calls (`pow`), they are bound by the strict laws of floating-point mathematics.

The analysis proves that while your manual hoisting in `loop.2.c` helped keep the code clean, **the substitution of division with reciprocal multiplication was the true driver of optimization** that the compiler could not perform on its own.
